## Summary

`taxes` is a local household tax workflow for ingesting documents, resolving facts, calculating withholding, managing tax rules, and exposing the workflow through CLI, MCP, and web surfaces.

It has two distinct trust layers:

- Household fact verification: raw documents -> extracted facts -> resolved facts -> verified facts -> return package.
- Managed rule provenance: official tax-rule sources -> extracted candidates -> reconciled values -> confirmed, discovered, or conflict-flagged rule data.

This repo is one of the strongest examples of high-stakes trust because wrong values can materially affect financial outcomes.

## Trust-Bearing Claims

- A document was imported from an allowed source path and stored with checksum identity.
- A document was classified as a specific form type.
- A fact was extracted from a document with a confidence score.
- A resolved fact won over competing candidates.
- A resolved fact requires human verification or was promoted automatically.
- A verified fact is safe to use in calculations.
- A return-package field is backed by citations, selected fact traces, rules, or assumptions.
- A prepared return differs from generated calculations by a known amount.
- A tax-rule value came from a specific official source URL, section, page, and snippet.
- A managed tax-rule value is confirmed, provisional, discovered, or conflict-flagged.
- A discovery workflow or manual override was used when deterministic source retrieval was blocked.

## Current Evidence And Validation

- Shared schemas distinguish `ExtractedFact`, `ResolvedFact`, and `VerifiedFact`.
- Fact resolution deduplicates candidates, ranks source types, compares confidence, tracks candidate sets, and removes verified facts when conflicts require review.
- Return packages include citations, selected fact traces, assumptions, reconciliation fields, driver sources, and review signals.
- Rules management tracks source availability, source URLs, parser kind, fallback usage, discovery output, warnings, and blocked sources.
- Managed rule metadata stores value statuses, citations, changelog entries, source page/section, confidence, raw snippets, and human-confirmation requirements.
- Value reconciliation flags conflicts when confirmed values differ from newly extracted source values.
- The repo also has `.veritas` governance/evidence, making it a strong Veritas proving ground.

## Missing, Stale, Disputed, Or High-Impact Claims

- Confidence is not the same as verification; medium-confidence and conflicting document facts need explicit review.
- Generated return values need traceable drivers, not only final numbers.
- Assumptions can be necessary but should not be hidden inside calculations.
- Rules data can be correct for one year and stale or blocked for another.
- Discovery-first rule sources need a different trust state than deterministic official URLs.
- Manual source overrides are sometimes necessary but must remain visible as provenance.
- A return package needs to communicate both what is ready and what still needs review.

## Veritas Fit

Taxes is a high Veritas fit because repo changes can weaken trust boundaries:

- If fact-resolution logic changes, tests should prove conflict handling, verification gating, and candidate ordering.
- If return-package generation changes, citation and review-signal tests should change.
- If managed-rules schemas or parsers change, source-registry and reconciliation tests should change.
- If web/admin rule surfaces change, browser proof lanes should run.
- `.veritas` can make these proof requirements local, explicit, and agent-facing.

## Surface Fit

Taxes is a high Surface fit because it already has claim/evidence/check/status semantics.

Surface should be able to represent:

- `tax-extracted-fact` claims from document parsing.
- `tax-resolved-fact` claims that are proposed and may need verification.
- `tax-verified-fact` claims promoted by user or system.
- `tax-return-field` claims backed by citations, calculation traces, and selected facts.
- `tax-assumption` claims that remain disputed or review-required.
- `tax-rule-value` claims with source URL, section, page, status, and changelog.
- `rules-source-availability` claims for official source retrieval, fallback, discovery, blocked, or manual states.

## Product Ideas

- High-stakes fact-verification adapter for document-heavy workflows.
- Return-package trust report that separates ready fields, assumptions, unresolved values, and discrepancies.
- Rules/source provenance report for compliance data that changes every year.
- Agent-readable "do not rely yet" surface for unresolved facts, blocked sources, and conflict-flagged values.
- Human review queue ordered by materiality, not only missingness.

## Foundation Lessons

- The durable trust boundary is raw extraction -> resolved candidate -> verified fact -> generated artifact.
- A generated artifact should carry its evidence chain, assumptions, and discrepancy trace.
- Official source management is its own trust workflow: source discovery, fallback, parsing, confirmation, and conflict detection.
- Surface should support values that are verified for one use but still require review for another.
- Materiality matters. Not every missing or disputed fact has equal product risk.
- Veritas and Surface reinforce each other: Veritas protects the code and rule-change workflow; Surface explains the domain trust state.
